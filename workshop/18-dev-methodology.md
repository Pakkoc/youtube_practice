# Claude Code 개발 방법론 가이드

> FSD, SDD, DDD — AI 시대의 소프트웨어 설계 원칙을 쉽게 이해하기

---

## 왜 이 문서가 필요한가?

Claude Code로 영상을 만드는 것은 **"도구를 쓰는 것"**입니다.
하지만 이 시스템을 **내 채널에 맞게 고치고, 스킬을 만들고, 확장**하려면 설계 원리를 알아야 합니다.

이 문서에서 다루는 3가지 방법론:

| 방법론 | 한 줄 요약 | 이 시스템에서의 역할 |
|--------|-----------|-------------------|
| **FSD** | 코드를 기능 단위로 쪼개는 폴더 구조 | 프로젝트의 뼈대 |
| **SDD** | 코딩 전에 명세서(스펙)를 먼저 쓴다 | 기능 추가 방식 |
| **DDD** | 현실 세계의 개념을 코드에 그대로 반영 | 데이터 모델 설계 |

> **비유:** 집을 지을 때 FSD는 **설계도(구조)**, SDD는 **시공 계획서(절차)**, DDD는 **방 이름표(용어 통일)**입니다.

---

## Part 1: FSD — Feature-Sliced Design

### 한 줄 정의

> 코드를 **기능(Feature) 단위**로 쪼개고, **레이어(층)**로 쌓는 폴더 구조 규칙

### 비유: 백화점 건물

백화점을 생각해 보세요:

```
5층  경영진 사무실     ←  전체 관리         = app/
4층  행사 기획팀       ←  여러 매장 조율    = pipelines/
3층  각 매장 (옷, 신발, 가전)  ←  독립 기능  = features/
2층  공용 창고 (상품 분류)     ←  데이터 정의 = entities/
1층  주차장, 엘리베이터        ←  공용 인프라 = shared/
```

**핵심 규칙: 위층은 아래층을 이용할 수 있지만, 아래층은 위층을 부르지 않는다.**

- 5층 경영진이 3층 매장에 지시할 수 있다 (O)
- 3층 매장이 5층 경영진에게 직접 명령할 수 없다 (X)
- 3층 신발매장이 3층 옷매장의 물건을 가져갈 수 없다 (X — 같은 층끼리도 직접 참조 금지)

### 이 시스템의 실제 구조

```
app/        → 입구 (CLI 명령어 정의, 설정 로드)
  ↓ 호출
pipelines/  → 관리자 (5가지 파이프라인 순서 제어)
  ↓ 호출
features/   → 실무자 (TTS, 슬라이드, B-roll, 자막 등 20개+)
  ↓ 참조
entities/   → 서류 양식 (Script, Slide, AudioClip 등 데이터 정의)
  ↓ 참조
shared/     → 공용 도구 (API 클라이언트, FFmpeg, 설정 스키마)
```

### 왜 이렇게 하나?

| 문제 | FSD가 해결하는 방법 |
|------|-------------------|
| "이 코드 어디 있지?" | features/generate_tts/ → TTS 관련 코드는 여기 |
| "이거 고치면 다른 데 터지나?" | 같은 층끼리 의존 안 하므로 영향 범위가 명확 |
| "새 기능 추가는?" | features/에 새 폴더 만들면 끝 |
| "누가 규칙 어겼는지?" | `import-linter`가 자동으로 잡아줌 |

### Feature 폴더 내부 구조

각 기능 폴더는 4개 파일로 구성됩니다:

```
features/generate_tts/
├── __init__.py    ← 외부에 공개하는 함수 (출입구)
├── model.py       ← 이 기능 전용 데이터 구조
├── api.py         ← 외부 서비스 호출 (ElevenLabs 등)
└── lib.py         ← 핵심 로직 (비즈니스 규칙)
```

> **비유:** 매장마다 **간판(init)**, **재고 목록(model)**, **외부 거래처 연락망(api)**, **매장 운영 매뉴얼(lib)**이 있는 것과 같습니다.

## Part 2: SDD — Specification-Driven Development

### 한 줄 정의

> **코딩을 시작하기 전에** 명세서(스펙)를 먼저 작성하고, 그 스펙대로 구현하는 방식

### 비유: 요리 레시피

```
❌ 일반적 방식: "일단 만들어보자" → 실패 → 수정 → 또 실패 → ...
✅ SDD 방식:   레시피 먼저 작성 → 재료 확인 → 순서대로 조리 → 맛 검증
```

### SDD의 3단계

```
[1] 명세서 작성 (뭘 만들 건지)
     ↓
[2] 구현 (명세서대로 코딩)
     ↓
[3] 검증 (명세서의 체크리스트로 확인)
```

### 이 시스템의 SDD: `/spec-writer`

이 시스템에는 SDD를 자동화하는 스킬이 있습니다:

```
/spec-writer
```

실행하면 **3개 파일**이 생성됩니다:

| 파일 | 역할 | 비유 |
|------|------|------|
| `spec.md` | 기능 명세서 — 뭘 만드는지 | 요리 완성 사진 |
| `plan.md` | 구현 계획 — 어떻게 만드는지 | 요리 레시피 |
| `acceptance.md` | 검증 기준 — 완성인지 확인하는 체크리스트 | 맛 평가표 |

### 왜 코드 전에 스펙을 쓰나?

| 문제 | SDD가 해결하는 방법 |
|------|-------------------|
| "이거 다 했나?" | acceptance.md 체크리스트로 확인 |
| "원래 뭘 하려고 했지?" | spec.md에 목적이 명시됨 |
| "왜 이렇게 만들었지?" | plan.md에 설계 근거가 기록됨 |
| "Claude Code가 엉뚱한 걸 만들었다" | 스펙이 있으면 방향을 잡아줌 |

### Claude Code에서의 SDD 활용

Claude Code에게 기능을 요청할 때도 SDD 원칙이 적용됩니다:

```
❌ "TTS 기능 좀 고쳐줘"
✅ "TTS에서 숫자를 한글로 변환하는 기능이 필요해.
    입력: '100만원' → 출력: '백만 원'.
    기존 tts_dictionary.yaml에 규칙을 추가하는 방식으로."
```

**구체적 스펙 = 정확한 구현.** 사람에게 일 시키는 것과 같습니다.

## Part 3: DDD — Domain-Driven Design

### 한 줄 정의

> 현실 세계의 **개념(도메인)**을 코드에 그대로 반영하는 설계 방식

### 비유: 병원 시스템

병원 시스템을 만든다면:

```
❌ 나쁜 설계:  data1, data2, item_list, process()
✅ DDD 설계:   Patient(환자), Doctor(의사), Prescription(처방전), Appointment(예약)
```

현실에서 쓰는 용어를 코드에서도 그대로 사용합니다.

### 이 시스템의 도메인 용어

이 영상 자동화 시스템에서 현실 개념이 코드에 어떻게 반영되는지:

| 현실 개념 | 코드 모델 (entities/) | 설명 |
|-----------|---------------------|------|
| 대본 | `Script` | 전체 영상 대본 |
| 문단 | `Paragraph` | 대본의 한 단락 |
| 슬라이드 | `Slide` | 한 장의 화면 디자인 |
| 음성 클립 | `AudioClip` | 문단 하나의 음성 파일 |
| 프로젝트 | `Project` | 하나의 영상 작업 단위 |
| 영상 | `Video` | 최종 출력물 |

### DDD의 핵심: 유비쿼터스 언어

**모든 사람이 같은 용어를 쓴다**는 원칙입니다:

```
기획자: "3번째 문단의 슬라이드를 수정해주세요"
코드:   project.paragraphs[2].slide.update(...)
Claude: "3번째 문단의 슬라이드를 수정했습니다"
```

기획자, 코드, AI가 모두 **"문단"**, **"슬라이드"**라는 같은 단어를 씁니다.
이렇게 하면 소통 오류가 줄어듭니다.

### 왜 DDD를 쓰나?

| 문제 | DDD가 해결하는 방법 |
|------|-------------------|
| "이 변수가 뭐지?" | 현실 용어니까 이름만 봐도 이해 |
| "어디에 코드를 추가하지?" | 도메인에 해당하는 entities/ 폴더에 |
| "Claude Code에게 뭐라고 말하지?" | 현실 용어 그대로 말하면 됨 |

## Part 4: 세 방법론의 조합

세 방법론은 각각의 질문에 답합니다:

```
DDD:  "이 시스템은 어떤 개념들로 구성되나?" → 용어 정의
FSD:  "코드를 어디에 놓을 것인가?"         → 폴더 구조
SDD:  "새 기능을 어떻게 추가할 것인가?"     → 작업 절차
```

### 실전 예시: "쇼츠에 워터마크 추가" 기능

**DDD 단계 — 개념 정의:**
```
새 개념: Watermark (워터마크)
  - 위치 (position): 우하단
  - 이미지 (image_path): 로고 파일 경로
  - 투명도 (opacity): 0.3
```

**FSD 단계 — 어디에 넣을지:**
```
features/add_watermark/     ← 새 기능 폴더 생성
├── __init__.py
├── model.py               ← Watermark 데이터 정의
├── lib.py                 ← FFmpeg 워터마크 합성 로직
└── api.py                 ← (외부 서비스 없으므로 생략)
```

**SDD 단계 — 스펙 먼저:**
```
/spec-writer
→ spec.md:        워터마크 기능 요구사항 정의
→ plan.md:        FFmpeg overlay 필터 사용 계획
→ acceptance.md:  "영상 우하단에 로고가 30% 투명도로 표시됨" 체크
```

## Part 5: CLAUDE.md — AI의 행동 규칙서

### CLAUDE.md란?

프로젝트 루트에 있는 `CLAUDE.md`는 **Claude Code의 행동 규칙서**입니다.

```
CLAUDE.md = "이 프로젝트에서 Claude Code는 이렇게 행동해라"
```

### 비유: 신입 직원 온보딩 문서

새 직원이 왔을 때 건네는 문서와 같습니다:

```
1. 우리 회사 코딩 컨벤션은 이렇다
2. 이 API는 이렇게 호출한다
3. 절대 하지 말아야 할 것 목록
4. 작업 완료 전 체크리스트
```

Claude Code는 **매 세션마다** 이 파일을 읽고 규칙을 따릅니다.

### CLAUDE.md의 구성 요소

| 섹션 | 역할 | 예시 |
|------|------|------|
| Project Overview | 프로젝트 설명 | "AI 영상 자동화 시스템" |
| Model & API Versions | 사용할 모델 지정 | "GPT-5 사용, Claude 모델 금지" |
| Architecture | 코드 구조 규칙 | "FSD 레이어 순서 준수" |
| Verification | 완료 기준 | "lint + test + FSD import 검증" |
| Commands | 실행 명령어 | "ruff check .", "pytest" |

### CLAUDE.md가 중요한 이유

```
CLAUDE.md 없는 Claude Code:
  → 매번 "어떤 모델 쓰지?", "폴더 구조가 뭐지?" 물어봄
  → 일관성 없는 코드 생성
  → 실수 반복

CLAUDE.md 있는 Claude Code:
  → 규칙을 이미 알고 있음
  → 일관된 코드 생성
  → 자체 검증까지 수행
```

### CLAUDE.md 작성 팁

```
✅ 좋은 규칙: "GPT-5 API는 max_completion_tokens를 사용한다 (max_tokens 금지)"
  → 구체적, 검증 가능

❌ 나쁜 규칙: "좋은 코드를 작성해라"
  → 모호함, 해석이 달라질 수 있음
```

**핵심 원칙:** CLAUDE.md의 규칙은 **구체적이고 검증 가능**해야 합니다.

## Part 6: Skills — 반복 가능한 절차의 자산화

### Skill이란?

**Skill = Claude Code에게 가르치는 반복 가능한 작업 절차**

```
Skill 없이:
  매번 "슬라이드 만들 때 이런 규칙으로, 이런 레이아웃으로, 이런 애니메이션으로..."
  → 매번 설명해야 함, 품질 들쭉날쭉

Skill 있으면:
  "/generate-video workshop-001"
  → 한 번에 모든 규칙이 적용됨, 일관된 품질
```

### CLAUDE.md vs Skills

| | CLAUDE.md | Skills |
|-|-----------|--------|
| **역할** | 전체 행동 규칙 | 특정 작업 절차 |
| **비유** | 회사 사규 | 업무 매뉴얼 |
| **범위** | 프로젝트 전체 | 기능별 |
| **수정** | 신중하게 (제안 후 승인) | 스킬 크리에이터로 자유롭게 |

### Skill의 구조

```
.claude/commands/generate-video/SKILL.md
├── 목적: "대본 → 완성 영상" E2E 파이프라인
├── 입력: script.txt 경로
├── 단계: TTS → 슬라이드 → B-roll → 렌더링 → 합성
├── 규칙: "B-roll 기본 활성화", "발음 사전 적용"
└── 출력: final_video.mp4
```

### 방법론과의 연결

```
CLAUDE.md (행동 규칙)          ← DDD 용어 + FSD 구조 규칙 명시
  ↓ 참조
Skills (작업 절차)              ← SDD 원칙으로 설계된 절차
  ↓ 실행
features/ (기능 코드)           ← FSD 구조에 맞춰 배치
  ↓ 사용
entities/ (데이터 모델)          ← DDD 도메인 모델
```

## Part 7: Claude Code에서의 실전 활용

### 일상적인 영상 제작 (도구 사용자)

대부분의 경우 방법론을 의식할 필요 없습니다:

```bash
# 대본 작성
"AI 자동화를 주제로 3분짜리 대본 써줘"

# 영상 생성
/generate-video workshop-001

# 수정
"3번 슬라이드 배경색 바꿔줘"
```

### 시스템 커스터마이징 (도구 제작자)

시스템을 고치거나 확장할 때 방법론이 빛납니다:

```
"새 TTS 백엔드를 추가하고 싶어"
→ DDD: TTSBackend 개념 정의
→ FSD: features/generate_tts/에 추가
→ SDD: /spec-writer로 스펙 먼저 작성
```

### 나만의 스킬 만들기

```
"우리 채널은 항상 인트로에 질문을 넣고,
 숫자 강조 슬라이드를 3개 이상 포함하고,
 아웃트로에 구독 유도 문구를 넣어"
→ 이 규칙을 Skill로 만들면 매번 자동 적용
```

## Part 8: 오픈소스 레퍼런스로 내 시스템 확장하기

### 왜 다른 프로젝트를 참고하나?

영상 자동화는 빠르게 발전하는 분야입니다. 비슷한 문제를 푸는 오픈소스 프로젝트를 분석하면 **내 시스템에 없는 아이디어**를 발견하고, 검증된 코드를 레버리지할 수 있습니다.

> **비유:** 같은 업종의 잘 되는 가게를 벤치마킹하는 것과 같습니다. 복사가 아니라, "저 가게는 저걸 왜 저렇게 하지?"를 분석하고 내 가게에 맞게 적용하는 것입니다.

### 레퍼런스 프로젝트: MoneyPrinterTurbo

[MoneyPrinterTurbo](https://github.com/harry0703/MoneyPrinterTurbo)는 "주제/키워드 하나로 쇼츠 영상을 자동 생성"하는 오픈소스 프로젝트입니다. GitHub에서 큰 관심을 받고 있는 레포지토리로, 이 워크숍 시스템과 같은 문제를 다른 방식으로 풀고 있습니다.

#### 파이프라인 비교

| 단계 | 이 시스템 (Youtube-Automation) | MoneyPrinterTurbo |
|------|-------------------------------|-------------------|
| **대본 생성** | Claude Code가 작성 / 직접 작성 | LLM API 자동 생성 (OpenAI, DeepSeek, Gemini 등 10개+ 지원) |
| **음성 합성** | ElevenLabs (고품질 1개) | Azure, Edge TTS 등 다수 |

### Claude Code로 레퍼런스 레포 분석하는 법

다른 오픈소스 프로젝트에서 유용한 코드를 찾아 내 시스템에 적용하는 실전 워크플로우입니다.

#### Step 1: 레포 클론 + 구조 파악

```bash
# 별도 폴더에 클론 (내 프로젝트와 분리)
cd ~/Desktop
git clone https://github.com/harry0703/MoneyPrinterTurbo.git
cd MoneyPrinterTurbo
```

Claude Code를 열고 구조를 파악합니다:

```
이 프로젝트의 전체 아키텍처를 분석해줘.
핵심 파이프라인 흐름과 주요 모듈을 정리해줘.
```

#### Step 2: 관심 기능의 코드 찾기

내 시스템에 없는 기능을 발견했다면, 해당 기능의 구현을 분석합니다:

```
이 프로젝트에서 BGM 자동 삽입 기능이 어떻게 구현되어 있는지 찾아줘.
- 어떤 파일에 있는지
- 입력/출력이 뭔지
- FFmpeg 명령어가 어떻게 구성되는지
핵심 로직만 정리해줘.
```

```
Pexels API로 영상을 검색하는 코드를 찾아줘.
- API 호출 방식
- 검색 키워드 생성 로직
- 영상 다운로드 + 크롭/리사이즈 처리
를 분석해줘.
```

#### Step 3: 내 시스템에 적용 설계

분석 결과를 가지고 내 프로젝트로 돌아와서 적용 계획을 세웁니다:

```
MoneyPrinterTurbo의 BGM 자동 삽입 로직을 분석한 결과를 바탕으로,
우리 시스템(Youtube-Automation)에 BGM 자동 삽입 기능을 추가하려면
어떻게 설계해야 하는지 /spec-writer로 명세를 작성해줘.
```

#### Step 4: SDD 원칙대로 구현

Part 2에서 배운 SDD 절차를 그대로 따릅니다:

```
[1] /spec-writer → 명세서 3개 파일 생성
[2] 명세서 검토 + 승인
[3] 구현 (FSD 구조에 맞춰 features/에 배치)
[4] acceptance.md 체크리스트로 검증
```

### 레퍼런스 분석 시 주의사항

```
✅ 좋은 접근:
  - "이 기능의 핵심 로직(알고리즘, API 호출 패턴)을 이해하고 내 구조에 맞게 새로 구현"
  - "FFmpeg 명령어 조합이나 API 파라미터 같은 도메인 지식을 참고"
  - "아이디어와 접근법을 벤치마킹"

❌ 나쁜 접근:
  - "코드를 통째로 복사해서 붙여넣기"
  - "라이선스를 확인하지 않고 사용"
  - "내 시스템의 아키텍처(FSD)를 무시하고 그냥 끼워넣기"
```

> **핵심:** 오픈소스 레퍼런스에서 가져올 것은 **코드가 아니라 아이디어와 도메인 지식**입니다. 구현은 반드시 내 시스템의 아키텍처(FSD)와 절차(SDD)에 맞춰 새로 합니다.

### 다른 참고할 만한 오픈소스 프로젝트

영상 자동화 분야의 오픈소스 생태계는 빠르게 성장하고 있습니다. Claude Code에게 다음과 같이 요청하면 최신 프로젝트를 탐색할 수 있습니다:

```
각 프로젝트의 핵심 기능과 우리 시스템과의 차이점을 정리해줘.
```
