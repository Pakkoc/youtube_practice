# /generate-script -- 통합 대본 생성

모든 대본 생성 경로를 하나의 커맨드로 통합.
- **Longform**: 기술 문서 -> 유튜브 영상 대본 (2-10분)
- **Shorts**: 소스 -> 쇼츠 대본 (45-60초)
- **CC**: Claude Code 교육 쇼츠 카드 -> 대본

---

## 포맷 자동 감지

인자를 파싱하여 경로를 결정:

```
--list                          -> CC 대기 카드 목록 출력 후 종료
--card-id cc-XXX 또는 cc-* 패턴 -> CC 경로
--format shorts                 -> Shorts 경로
--format cc                     -> CC 경로
그 외                           -> Longform 경로 (Step 2에서 쇼츠/숏폼/풀영상 세부 결정)
```

인자 파싱: `$ARGUMENTS`에서 프로젝트명, --format, --card-id, --list 플래그를 추출.

---

## Step 1: 프로젝트 + 소스 로드

### Longform / Shorts 경로

1. `projects/$PROJECT/` 디렉토리 확인 (없으면 생성)
2. 소스 문서 탐색 -- 아래 순서로 시도:
   - `projects/$PROJECT/source.md` 또는 `source.txt` -> Read
   - 사용자가 URL을 제공했으면 -> WebFetch로 로드
   - 위 둘 다 없으면 -> 안내:
     ```
     소스 문서를 아래 중 하나로 제공해주세요:
     1. projects/$PROJECT/source.md 파일로 저장
     2. URL 붙여넣기
     3. 텍스트 직접 붙여넣기
     ```
3. 기존 `projects/$PROJECT/script.txt`가 있으면 AskUserQuestion으로 덮어쓰기 여부 확인

### CC 경로

1. `cc-content/cards.json` 로드 -> Read
2. `--card-id`로 지정된 카드 매칭 (없으면 `$PROJECT`에서 cc-XXX 패턴 추출)
3. `--list`이면 `status: "pending"` 카드 목록을 테이블로 출력 후 **종료**
4. `projects/$PROJECT/` 디렉토리 확인 (없으면 생성)

---

## Step 2: 콘텐츠 분석 (경로별 분기)

### Longform 경로

1. 소스에서 핵심 메시지 3-5개 추출
2. AskUserQuestion으로 포맷 확인:

| 포맷 | 길이 | 문단 수 | 핵심 메시지 |
|------|------|---------|------------|
| **쇼츠** | 45-60초 | 4-6개 | 핵심 1-2개만 |
| **숏폼** | 2-3분 | 8-15개 | 핵심 3-4개 |
| **풀영상** | 5-10분 | 20-40개 | 포괄적 |

3. 사용자가 **쇼츠**를 선택하면 -> Shorts 워크플로로 전환 (Step 3 Shorts 경로)
4. 사용자의 핵심 메시지 수정/추가/삭제 피드백 수용

### Shorts 경로

1. `docs/shorts/hook-strategy.md` 로드 -> Read
2. 소스에서 핵심 메시지 1-3개 추출
3. 콘텐츠 유형 분류:

| 유형 | 설명 | 예시 |
|------|------|------|
| 정보/뉴스 | 새 소식, 업데이트 | "GPT-5 출시" |
| 하우투/팁 | 방법론, 꿀팁 | "Claude Code 생산성 2배" |
| 리뷰/비교 | 제품 비교, 장단점 | "Cursor vs Claude Code" |
| 의견/경험 | 개인 관점, 후기 | "3개월 사용 후기" |
| 데모/쇼케이스 | 실습, 시연 | "10분만에 앱 만들기" |

4. **시청자 공감 분석** — 훅 작성 전 필수 단계. 패턴보다 사람이 먼저다:
   - **타겟 페르소나**: 이 주제를 검색하거나 클릭할 사람은 누구인가? (역할, 상황, 현재 고민)
   - **숨은 욕구 3가지**: ① 가장 듣고 싶어하는 말 ② 가장 두려워하는 것 ③ 가장 궁금한 것
   - **공감 워딩 3개**: 타겟 시청자가 "맞아 맞아!" 또는 "어? 그거 나인데?" 할 구체적 표현

   > 예시 — 주제: "RAG가 죽었다는 말의 진실"
   > - 페르소나: RAG 프로젝트 경험이 있는 AI 엔지니어, 최근 "RAG is dead" 글을 보고 불안한 사람
   > - 욕구: ① "내가 해온 RAG 작업이 헛되지 않았다" ② "방향을 잘못 잡고 있는 건 아닐까" ③ "그래서 앞으로 어떻게 해야 하는데?"
   > - 공감 워딩: "요즘 RAG 죽었다는 말 보면 불안하시죠?", "죽은 건 RAG가 아닙니다", "당신의 RAG 경험이 오히려 무기입니다"

5. 분석 결과 (유형 + 페르소나 + 공감 워딩) 테이블 제시 -> AskUserQuestion으로 확인
   - 사용자가 페르소나나 공감 워딩을 수정할 수 있음

### CC 경로

1. 카드에서 `feature_name`, `content_format`, `dev_term`, `plain_term` 추출
2. 레퍼런스 수집 (WebSearch + WebFetch):
   - 공식 문서: `"Claude Code" "{feature_name}" site:docs.anthropic.com`
   - 커뮤니티: `"Claude Code" "{feature_name}" site:reddit.com`
   - 기저 개념 연구 (dev_term이 일반 프로그래밍 용어인 경우)
3. `cc-content/reference/{card_id}.md`에 수집 결과 캐싱 (이미 존재하면 Read로 재사용)

## Step 3: 대본 작성 (경로별)

Claude Code가 직접 한국어 구어체 대본을 작성 (API 호출 아님).

### 공통 원칙 (모든 경로)

1. **구어체**: "~거든요", "~인데요", "~잖아요" 등 자연스러운 종결어미
2. **영어 원형 보존**: 브랜드명/기술 용어는 영어 그대로 (Claude, API, Docker)
3. **숫자 아라비아**: "30%", "100만", "2배" (TTS가 한글 발음으로 읽음)
4. **문단 = 씬**: 빈 줄로 구분된 각 문단이 하나의 슬라이드 + TTS 단위

**톤 예시:**
```
X: "해당 기술은 인공지능 분야에서 매우 중요한 역할을 수행합니다."
O: "이 기술이 왜 중요하냐면요, AI 분야에서 게임 체인저거든요."

X: "다음으로 살펴볼 내용은 성능 벤치마크입니다."
O: "그러면 성능은 어떨까요? 벤치마크를 한번 보겠습니다."
```

### Longform 전용 (숏폼/풀영상)

- **첫 문단**: 후킹 -- 질문, 놀라운 사실, 문제 제기
- **본문**: 핵심 메시지를 논리적 순서로 전달. 각 문단이 하나의 포인트
- **마지막**: CTA(구독/좋아요 유도) 또는 핵심 요약
- **문단 길이**: 30-80자 (TTS 5-15초). 100자 초과 시 분할, 15자 미만 시 병합 검토
- **상보성**: 슬라이드 시각자료를 보완하는 내레이션. 시각자료 내용을 그대로 반복하지 말 것

### Shorts 전용

1. **훅 전략 제안**: Step 2의 **공감 워딩을 기반으로** 3개 훅 후보 작성 (`docs/shorts/hook-strategy.md` 패턴 참조)
   - 훅에 공감 워딩이 자연스럽게 녹아들어야 함. 패턴만 적용한 정보 전달형 훅 지양
   - 시청자의 숨은 욕구(듣고 싶은 말/두려움/궁금증) 중 하나를 건드리는 것이 핵심
2. AskUserQuestion으로 훅 전략 선택
3. **4-6문단** (45-60초 목표):
   - 1문단: 훅 (15-25자, 공감 워딩 + 선택한 전략 적용)
   - 2-4문단: 핵심 포인트 1-2개 (배경 설명 제거, 결론 먼저, 예시 1개만)
   - 마지막: 결론 또는 CTA
4. **훅 타이틀 초안** 2-3개 작성 (`docs/shorts/hook-patterns-70.md` 참고)
5. 총 글자 수 150-280자 목표

### CC 전용

1. `cc-content/docs/script-guide.md` 로드 -> Read (없으면 skip)
2. `content_format`에 따른 포맷별 프롬프트 적용:
   - **translator**: dev_term <-> plain_term 번역 구조
   - **flow**: 단계별 흐름도 설명 구조
   - **situation**: 상황극 시나리오 구조
3. Step 2에서 수집한 레퍼런스 기반으로 깊이 있는 설명
4. CC 품질 기준: 비개발자 이해 가능성, plain_term 활용도, 실생활 비유 포함

#### CC 훅 작성법 (첫 문단)

CC 대본의 첫 문단은 **시청자의 실제 경험/감정에서 시작**해야 한다. 기능 설명이나 개발자 맥락으로 시작하면 안 된다.

**CC 훅 공식 (3가지 중 택1):**

| 공식 | 구조 | 예시 |
|------|------|------|
| **좌절 공감형** | "[누구나 겪는 AI 사용 불편] + [공감 질문]" | "AI한테 매번 같은 설명 반복하는 거, 진짜 지치지 않으세요?" |
| **만약에형** | "만약에 [놀라운 가능성]이면 어떨까요?" | "만약에 명령어 하나로 프로젝트 전체를 파악할 수 있다면요?" |
| **상황 묘사형** | "[구체적 상황 묘사] + [불편/답답함]" | "AI한테 파일 수정 시켰더니 매번 허락을 구해요. 한 시간에 100번 넘게 허락 버튼을 누르고 있는 겁니다." |

**CC 훅 금지:**
- ❌ 기능명으로 시작 ("Claude Code에서 X 기능을 쓰면...")
- ❌ 개발자만 공감하는 맥락 ("프론트엔드 코드를 작성하는데...")
- ❌ 추상적 가치 제안 ("생산성이 2배 올라갑니다")
- ❌ **첫 문단에 영어 약어/기술 용어 사용 절대 금지** (PR, API, CI/CD, deploy, branch 등). 훅은 100% 한국어 일상 표현으로만 구성. 기술 용어는 2번째 문단 이후에 plain_term과 함께 도입.

#### CC 비개발자 용어 변환 규칙

**모든 개발 용어는 등장 즉시 쉬운 설명이 따라야 한다.** `dev_term`은 카드의 `plain_term`으로 번역하고, 본문에 등장하는 추가 기술 용어도 반드시 부연한다.

**변환 패턴:**
- `"[dev_term], 쉽게 말하면 [plain_term]인데요"` — 자연스러운 구어체 삽입
- `"[dev_term]이라는 게 있는데요, [비유]랑 같은 거예요"` — 비유 직결

**자주 실수하는 미설명 용어 (반드시 변환):**
| 원어 | 비개발자 표현 |
|------|------------|
| PR (Pull Request) | 코드 합치기 요청 |
| branch | 작업 복사본 |
| commit | 저장 기록 |
| deploy | 실제 서비스에 반영 |
| CI/CD | 자동 검사/반영 시스템 |
| workflow | 자동화 설정 |
| 프론트엔드/백엔드 | 화면 쪽/서버 쪽 (또는 실생활 비유로 대체) |

5. **마무리 패턴 다양화 (필수)**: 아래 5가지 CC 마무리 패턴 중 하나를 선택. **직전 에피소드와 같은 패턴 사용 금지.**

   | # | 마무리 패턴 | 공식 | 예시 |
   |---|-----------|------|------|
   | 1 | **행동 유도형** | "지금 바로 [구체적 행동]" | "지금 바로 /init 한번 쳐보세요." |
   | 2 | **한 줄 요약형** | "딱 하나만 기억하세요: [핵심]" | "딱 하나만 기억하세요. 50% 넘으면 /compact입니다." |
   | 3 | **반전/떡밥형** | "근데 진짜 [X]는 따로 있습니다" | "근데 진짜 강력한 조합은 따로 있습니다. 다음 영상에서요." |
   | 4 | **질문 던지기형** | "여러분은 [선택지] 어떤 쪽이세요?" | "여러분은 자동승인파세요, 하나하나 확인파세요?" |
   | 5 | **실생활 연결형** | "[비유]랑 같은 거예요" | "핸드폰 정리 안 하면 느려지는 것처럼, AI도 똑같습니다." |

   > **CTA 규칙:** "이런 팁이 도움이 됐다면 이 영상에 좋아요 눌러주세요"처럼 독립된 CTA 문장으로 분리하는 것은 금지. 좋아요/구독 유도는 마무리 패턴의 핵심 메시지에 자연스럽게 녹여서 사용.
   > - ✅ "지금 바로 /init 쳐보세요. 도움이 됐다면 좋아요 한 번이면 큰 힘이 됩니다."
   > - ✅ "여러분은 자동승인파세요? 댓글로 알려주세요."
   > - ❌ "정리하면 X입니다. 이런 팁이 도움이 됐다면 이 영상에 좋아요 눌러주세요."

## Step 4: 대본 검수 (사용자 승인 게이트)

**모든 경로 공통:**

1. 대본 전문 표시 + 메타 정보:
   ```
   대본 생성 완료

   총 N개 문단 | 약 X자 | 예상 길이 Y분 Z초
   (한국어 TTS 기준 1초당 약 4-5자, 총 글자 수 / 4.5로 추정)

   --- 대본 전문 ---
   [대본 전체 텍스트]
   --- 끝 ---
   ```

2. AskUserQuestion 승인:
   - **"이대로 저장"** -> Step 5로 진행
   - **"수정 요청"** -> 피드백 반영 후 재작성, 다시 Step 4
   - **"직접 수정할게요"** -> `script_draft.txt`에 임시 저장, 사용자 편집 후 재요청 시 `script.txt`로 이동

### Shorts 추가 검수

훅 타이틀 함께 제시하고, 품질 체크리스트 확인:
- [ ] 훅에 공감 워딩이 녹아있는가 (단순 패턴 적용이 아닌 시청자 감정 연결)
- [ ] 훅 길이 15-25자
- [ ] 문단 수 4-6개
- [ ] 총 글자 수 150-280자
- [ ] 심리 트리거 포함 (호기심/놀라움/긴급성)
- [ ] 배경 설명 없이 바로 핵심

### CC 추가 검수

- [ ] 비개발자가 이해 가능한 설명
- [ ] plain_term이 자연스럽게 활용됨
- [ ] 공식 문서 정보가 정확히 반영됨

## Step 5: 발음 사전 스캔 + 업데이트

**모든 경로 공통.**

### 5a. 영어 용어 추출

대본에서 영어 단어/브랜드명 자동 추출 (정규식: 연속 영문자 2자+, 대소문자 불문).
일반 영단어(the, is, and 등)는 제외.

### 5b. 사전 대조

`config/tts_dictionary.yaml` Read -> `tts_pronunciation` 키 아래 등록 용어와 대조.

### 5c. 미등록 용어 보고

미등록 용어가 있으면 테이블로 보고:
```
발음 사전 스캔 결과

| 용어 | 제안 발음 | 사전 상태 |
|------|----------|----------|
| Claude | 클로드 | 등록됨 |
| worktree | 워크트리 | 미등록 |
| Deno | 디노 | 미등록 |
```

미등록 없으면: `모든 영어 용어가 발음 사전에 등록되어 있습니다.`

### 5d. 사용자 승인 후 업데이트

미등록 용어가 있을 때 AskUserQuestion:
- **"제안대로 추가"** -> `config/tts_dictionary.yaml`의 `tts_pronunciation:` 섹션에 추가 (알파벳 순서)
- **"발음 수정 후 추가"** -> 사용자 수정 발음으로 추가
- **"추가하지 않음"** -> skip

> YAML 형식: 기존 파일 형식 그대로. key=원형 영어, value=한글 발음. 들여쓰기 2칸.
