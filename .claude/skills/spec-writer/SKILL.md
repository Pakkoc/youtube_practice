# SDD Spec Writer

기능 구현 전에 3개의 명세 파일을 생성하여 **요구사항 → 계획 → 인수 기준**을 확정한다.
세션이 끊겨도 맥락이 보존되고, 구현 범위가 명확해진다.

## When to Use

- 사용자가 새 기능 구현을 요청할 때
- 기존 기능의 의미 있는 변경/확장이 필요할 때
- 복잡한 리팩토링이나 마이그레이션 계획이 필요할 때
- **예외**: 단순 버그 수정, 한 파일 수정, 설정 변경 등 trivial한 작업은 SPEC 없이 진행

## Workflow

### Phase 1: 요구사항 수집

사용자의 요청을 분석하고 아래를 파악한다:

1. **핵심 기능**: 무엇을 하는 기능인가?
2. **FSD 레이어 매핑**: 어떤 레이어/슬라이스에 영향을 주는가?
3. **도메인 엔티티**: 관련 엔티티(entities/)가 있는가? 새로 만들어야 하는가?
4. **외부 의존성**: API, DB, 라이브러리 등 외부 시스템이 필요한가?

요구사항이 모호하면 구현 전에 반드시 질문한다. 다만 사용자가 충분한 정보를 제공했으면 바로 Phase 2로 진행한다.

### Phase 2: 3파일 생성

`specs/[SPEC-ID]/` 디렉토리에 3개 파일을 생성한다.
SPEC-ID는 kebab-case 동사+명사 (예: `create-order`, `add-payment-webhook`).

---

#### File 1: `spec.md` — 요구사항 정의

```markdown
# [SPEC-ID] — [기능명]

## 배경
[왜 이 기능이 필요한지, 어떤 문제를 해결하는지]

## 요구사항 (EARS 형식)

### 핵심 요구사항
- **[REQ-01]** 시스템은 [기능]해야 한다.
- **[REQ-02]** WHEN [이벤트], THEN [동작]해야 한다.
- **[REQ-03]** WHILE [상태]인 동안, [동작]해야 한다.

### 금지사항
- **[REQ-N01]** 시스템은 [금지 동작]하면 안 된다.

### 선택사항
- **[REQ-O01]** 가능하다면, [선택 기능]해야 한다.

## 제약 조건
- [기술적 제약, 성능 요구, 호환성 등]

## FSD 레이어 매핑

| 레이어 | 슬라이스 | 변경 유형 |
|--------|----------|-----------|
| entities/ | [domain]/ | 신규 생성 / 수정 |
| features/ | [action]/ | 신규 생성 / 수정 |
| shared/ | [util]/ | 수정 |

## 의존성
- 내부: [기존 슬라이스 참조]
- 외부: [API, 라이브러리]
```

**EARS 형식 가이드:**

| 유형 | 패턴 | 용도 |
|------|-------|------|
| Ubiquitous | "시스템은 ~해야 한다" | 항상 적용되는 규칙 |
| Event-driven | "WHEN ~하면, THEN ~해야 한다" | 이벤트 반응 |
| Event+Condition | "WHEN ~하면, IF ~라면, THEN ~해야 한다" | 조건부 이벤트 |
| State-driven | "WHILE ~인 동안, ~해야 한다" | 상태 유지 동작 |
| Unwanted | "시스템은 ~하면 안 된다" | 금지사항 |
| Optional | "가능하다면, ~해야 한다" | 선택적 기능 |

요구사항 ID는 `REQ-01`(핵심), `REQ-N01`(금지), `REQ-O01`(선택)으로 구분한다.
모호한 표현("적절히", "빠르게") 대신 구체적 수치나 조건을 사용한다.

#### File 2: `plan.md` — 구현 계획

```markdown
# [SPEC-ID] 구현 계획

## 기술 스택
- [사용할 라이브러리, 프레임워크, 패턴]

## 파일 구조 계획

### 신규 생성
- `entities/[domain]/model/[name].py` — [역할]
- `features/[action]/api/[name].py` — [역할]
- `features/[action]/lib/[name].py` — [역할]

### 수정
- `shared/api/[name].py` — [변경 내용]

## 작업 분해

| # | TODO | 위험도 | 요구사항 |
|---|------|--------|----------|
| 1 | [작업 설명] | LOW | REQ-01 |
| 2 | [작업 설명] | MEDIUM | REQ-02 |
| 3 | [작업 설명] | HIGH | REQ-03 |

## 위험 분석

| 위험 | 영향 | 완화 전략 |
|------|------|-----------|
| [위험 설명] | HIGH | [구체적 대응책] |

## 의존성 분석
- [기존 코드 영향 범위]
- [FSD import 규칙 준수 확인 포인트]
```

**위험도 기준:**
- **LOW**: 기존 패턴 재사용, 단일 파일 변경
- **MEDIUM**: 새로운 패턴 도입, 복수 슬라이스 변경, 외부 API 연동
- **HIGH**: 기존 인터페이스 변경, 데이터 마이그레이션, 성능 민감 구간 → 구현 전 사용자 확인 필수

---

#### File 3: `acceptance.md` — 인수 기준

```markdown
# [SPEC-ID] 인수 기준

## 시나리오

### [REQ-01] [요구사항 요약]

**Scenario 1: [정상 케이스]**
- **Given** [사전 조건]
- **When** [사용자 행동 / 시스템 이벤트]
- **Then** [기대 결과]

**Scenario 2: [에러 케이스]**
- **Given** [사전 조건]
- **When** [예외 상황]
- **Then** [에러 처리 기대 결과]

## 엣지 케이스
- [ ] [경계값, 빈 입력, 동시성 등]

## 품질 게이트
- [ ] `ruff check .` 통과
- [ ] `ruff format --check .` 통과
- [ ] `mypy` 타입 체크 통과 (해당 시)
- [ ] `npx tsc --noEmit` 통과 (TSX 변경 시)
- [ ] `lint-imports` FSD 규칙 통과
- [ ] `pytest` 관련 테스트 통과
- [ ] 신규 코드 테스트 커버리지 확보

## 성능 기준 (해당 시)
- [응답 시간, 메모리 사용량 등 구체적 수치]
```

**Given/When/Then 작성 원칙:**
- Given: 검증 가능한 사전 상태 (DB 레코드 존재, 파일 존재 등)
- When: 단일 행동 (함수 호출, API 요청, 이벤트 발생)
- Then: 검증 가능한 결과 (반환값, 상태 변화, 부수 효과)
- 하나의 Scenario에는 하나의 When만 포함

---

### Phase 3: 리뷰 및 확정

3파일 생성 후:

1. **요약 제시**: 핵심 요구사항, 주요 작업, HIGH 위험 항목을 간단히 요약
2. **HIGH 위험 항목 명시**: "이 작업은 HIGH 위험으로 분류됩니다. 진행할까요?"
3. **사용자 확인 후 구현 시작**: 사용자가 승인하면 plan.md의 TODO 순서대로 구현 진행

## FSD 코드 배치 판단 트리

새 코드의 위치를 결정할 때 아래 순서로 질문한다:

```
비즈니스 로직 없는 순수 유틸? → shared/
    ↓ No
특정 도메인의 규칙/모델? → entities/[domain]/
    ↓ No
여러 페이지에서 재사용? → features/[action]/
    ↓ No
특정 페이지 전용? → pages/[page]/
```

## DDD 체크리스트 (entities/ 작업 시)

entities/ 슬라이스를 신규 생성하거나 수정할 때 아래를 확인한다:

- [ ] Aggregate Root 식별: 외부에서 접근하는 진입점 엔티티
- [ ] Value Object 분리: 불변이고 값으로만 비교되는 객체
- [ ] Repository 인터페이스: `model/`에 추상화, `api/`에 구현
- [ ] 도메인 모델에 프레임워크 import 없음 (순수 Python/TS)
- [ ] 비즈니스 규칙은 모델 내부에서 검증

## 주의사항

- **과도한 명세 금지**: 명세는 구현을 안내하는 도구이지, 그 자체가 목적이 아니다. 1~2시간 이상 명세에 투자하지 말 것.
- **SPEC-ID 충돌 방지**: 기존 `specs/` 디렉토리를 확인하여 중복 ID 방지.
- **점진적 보완**: 구현 중 발견된 새 요구사항은 acceptance.md에 추가하되, spec.md를 대폭 재작성하지 않는다.
- **언어**: 본문은 한국어, 코드 식별자와 기술 용어는 영어.
